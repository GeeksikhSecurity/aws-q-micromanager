# Contributing to AWS Q MicroManager

Thank you for your interest in contributing to AWS Q MicroManager! This document provides guidelines and instructions for contributing to this project.

## Code of Conduct

Please read and follow our [Code of Conduct](CODE_OF_CONDUCT.md).

## How to Contribute

### Reporting Issues

- Check if the issue has already been reported
- Use the issue template to provide all necessary information
- Include clear steps to reproduce the issue
- Specify your environment details

### Submitting Changes

1. Fork the repository
2. Create a new branch for your changes
3. Make your changes following our coding standards
4. Write or update tests as needed
5. Update documentation to reflect your changes
6. Submit a pull request

### Pull Request Process

1. Update the README.md or documentation with details of changes
2. Increase version numbers in any examples files to the new version
3. You may merge the Pull Request once you have the sign-off of at least one reviewer

## Prompt Development Guidelines

When creating or modifying prompts:

1. Keep prompts focused on specific roles and capabilities
2. Use clear, concise language
3. Include examples where helpful
4. Test prompts with AWS Q Developer before submitting
5. Consider token efficiency and response quality

## AI-Assisted Contributions

### Using AI Tools for Development

This project embraces AI-assisted development. When using AI tools:

1. **Transparency**: Clearly document AI-assisted contributions
2. **Attribution**: Follow the attribution guidelines in `docs/AI_ANALYSIS_ATTRIBUTION.md`
3. **Validation**: Always review and validate AI-generated content
4. **Integration**: Ensure AI contributions align with project goals

### AI Contribution Guidelines

#### **For AI-Generated Code**
- Include AI tool information in commit messages
- Add comments indicating AI-assisted sections
- Ensure code follows project standards and passes tests
- Example commit message: `feat: add testing framework (AI-assisted with Claude Sonnet 4)`

#### **For AI-Generated Documentation**
- Add attribution section to AI-generated documents
- Include generation date and AI system information
- Validate technical accuracy before submission
- Example attribution: `*Generated by Claude Sonnet 4 on [date] - human reviewed and validated*`

#### **For AI-Generated Analysis**
- Save AI analysis in `docs/ai-analysis/` directory
- Include methodology and scope documentation
- Add human review and validation notes
- Cross-reference with project requirements

### AI Tool Attribution Format

When documenting AI contributions:

```markdown
## AI Contribution Attribution

**AI System**: [Tool Name and Version]
**Provider**: [Company/Organization]
**Contribution Date**: [YYYY-MM-DD]
**Contribution Type**: [Analysis/Code/Documentation/Other]
**Human Review**: [Reviewer Name] on [Date]
**Validation Status**: [Approved/Modified/Rejected]
```

## Development Standards

### Code Quality
- Follow language-specific style guides
- Include proper error handling and logging
- Write unit tests for critical functionality
- Document complex logic with clear comments

### Testing Requirements
- All new features must include tests
- Maintain minimum 80% test coverage
- Include both unit and integration tests
- Test prompts with AWS Q Developer

### Documentation Standards
- Update relevant documentation for all changes
- Include examples where helpful
- Maintain clear, concise writing
- Follow markdown best practices

### Security Guidelines
- Never include hardcoded credentials or secrets
- Follow AWS security best practices
- Validate all user inputs
- Use secure coding practices

## Review Process

### For Human Contributors
1. Code review by at least one maintainer
2. All tests must pass
3. Documentation must be updated
4. Security review for sensitive changes

### For AI-Assisted Contributions
1. Human validation of AI-generated content
2. Technical accuracy verification
3. Alignment with project goals assessment
4. Integration testing with existing codebase

### Review Checklist
- [ ] Code follows project standards
- [ ] Tests are included and passing
- [ ] Documentation is updated
- [ ] AI contributions are properly attributed
- [ ] Security considerations are addressed
- [ ] Performance impact is acceptable

## Getting Started

### For New Contributors
1. Read the [README.md](README.md) and [INSTALLATION.md](INSTALLATION.md)
2. Review the [Repository Analysis](docs/REPOSITORY_REVIEW_AND_RECOMMENDATIONS.md)
3. Check the [Implementation Roadmap](docs/roadmap/IMPLEMENTATION_ROADMAP.md)
4. Start with issues labeled "good first issue"

### For AI-Assisted Development
1. Review [AI Analysis Attribution](docs/AI_ANALYSIS_ATTRIBUTION.md)
2. Understand the project's AI collaboration standards
3. Set up your development environment
4. Start with small, well-defined tasks

### Development Environment Setup
```bash
# Clone the repository
git clone [repository-url]
cd aws-q-micromanager

# Install development dependencies
pip install -r requirements-dev.txt

# Install pre-commit hooks
pre-commit install

# Run initial tests
pytest tests/
```
